Project Overview
Rustyjack is a Linux-first (Pi OS) wireless operations tool built for a Raspberry Pi Zero 2 W with a Waveshare 128×128 SPI LCD HAT (ST7735S) and an Ethernet HAT. The UI is a Rust binary (`rustyjack-ui`) that runs as a systemd service (`rustyjack.service`) with root privileges, enabling GPIO access, network control (`CAP_NET_ADMIN`), and wireless management via the companion crates `rustyjack-core`, `rustyjack-evasion`, and `rustyjack-wireless`.

Hardware Integration (Pi Zero 2 W + Waveshare 1.44" LCD HAT)
- Display (SPI, ST7735S), BCM pins (WAVESHARE_PINS.md):
  - SCLK: GPIO11 (Pin 23), MOSI: GPIO10 (Pin 19), CS: GPIO8 (Pin 24)
  - DC: GPIO25 (Pin 22), RST: GPIO27 (Pin 13), Backlight: GPIO24 (Pin 18)
  - Power: 3.3V (Pin 1/17), GND (Pin 6/9/14/20/25/30/34/39)
- Input (joystick + buttons), BCM pins:
  - Joystick: UP=6, DOWN=19, LEFT=5, RIGHT=26, PRESS=13
  - Buttons: KEY1=21, KEY2=20, KEY3=16
- Pull-ups: `/boot/firmware/config.txt` line `gpio=6,19,5,26,13,21,20,16=pu` is added by the installers to stabilize inputs.
- Display rotation: defaults to landscape via env `RUSTYJACK_DISPLAY_ROTATION=landscape` in the systemd service.
- Backlight control: togglable with `gpioset gpiochip0 24=1` (on) / `0` (off).

Built-in Wireless (Pi Zero 2 W)
- Chipset: Cypress/Infineon CYW43436 (2.4 GHz 802.11b/g/n, single-stream HT20, ~72 Mbps link). No 5 GHz.
- Driver capabilities: managed client and limited AP mode only. Stock driver does not support monitor or injection; deauth/Evil Twin/handshake capture require an external USB Wi-Fi adapter that supports monitor + injection (e.g., ath9k/ath9k_htc, rtl8812au with proper driver). Channel coverage is 2.4 GHz only.

Process Model and Service
- Installer copies `rustyjack-ui` to `/usr/local/bin` and creates `/etc/systemd/system/rustyjack.service` with `User=root`, `WorkingDirectory=$PROJECT_ROOT`, and `Environment=RUSTYJACK_DISPLAY_ROTATION=landscape`.
- Core logic is in `rustyjack-core` (CLI/ops), `rustyjack-evasion` (MAC/TX power/passive), `rustyjack-wireless` (nl80211, frames, injection).

UI Input Mapping (rustyjack-ui/src/app.rs)
- `map_button` maps raw GPIO input to actions:
  - Up/Down → selection
  - Left → back
  - Right/Select → accept
  - Key1 → refresh
  - Key2 → main menu
  - Key3 → reboot confirmation
- Menus use `choose_from_menu` to render lists and wrap selection with offset.

Display Driver (rustyjack-ui/src/display.rs)
- Uses `st7735-lcd` over SPI with GPIO-controlled DC/RST/BL lines:
  - DC line: `get_line(25)` (GPIO25)
  - RST line: `get_line(27)` (GPIO27)
  - Backlight line: `get_line(24)` (GPIO24)
- Draws dialogs/menus/progress using embedded-graphics primitives on a 128×128 canvas.

Config and Persistence (rustyjack-ui/src/config.rs)
- `gui_conf.json` stores pin defaults, color scheme, paths, and settings:
  - Active interface (`settings.active_network_interface`)
  - Target network/BSSID/channel (`settings.target_network`, `target_bssid`, `target_channel`)
  - MAC randomization flags and stored MACs (`original_mac`, `current_mac`)
  - Passive mode, TX power level

Network Scan and Target Selection (rustyjack-ui/src/app.rs:1535-1660)
- Wi-Fi scan: `fetch_wifi_scan` dispatches `Commands::Wifi(WifiCommand::Scan)` via `rustyjack-core`; results are shown as SSID buttons.
- On selection, `handle_network_selection` displays SSID, BSSID, channel, signal, encryption, and offers:
  - Connect (uses saved profile if present via `WifiProfileCommand::Connect`)
  - Set Target (writes target SSID/BSSID/channel to config and persists)
- Target is required by attacks; missing BSSID/channel is blocked later with user messaging.

Attack/Recon Flows and Target Usage
- Deauth (rustyjack-ui/src/app.rs:1610-1760): validates `target_bssid` and `target_channel`, then dispatches `WifiDeauthArgs { bssid, ssid, interface, channel, duration, packets, client=None, continuous=true, interval=1 }`. Displays progress and results (handshake capture, packets, log path).
- Evil Twin (app.rs:1810-1890): requires target SSID/BSSID/channel and interface; calls `WifiEvilTwinArgs { ssid, target_bssid, channel, interface, duration=300, open=true }`.
- PMKID Capture (app.rs:1940-2025): can target the configured SSID/BSSID/channel or run passive mode; uses `WifiPmkidArgs { interface, bssid, ssid, channel, duration }`.
- Probe Sniff (app.rs:1892-1940): uses active interface and channel hopping (`channel: 0`) with `WifiProbeSniffArgs`.
- Handshake cracking and loot viewing are wired to files in `loot/Wireless`.

MAC Address Management
- UI MAC randomize/restore uses `rustyjack-evasion::MacManager` (secure RNG, locally administered, unicast). Vendor OUI preference is derived from the current interface OUI via `VendorOui::from_oui`; falls back to random local-admin MACs.
- After MAC change, UI triggers `dhclient -r && dhclient` and signals Wi-Fi reconnect via `wpa_cli reconnect` or `nmcli device reconnect`.
- Installers ensure presence of `ip` (iproute2), `dhclient` (isc-dhcp-client), `nmcli` (network-manager), and `wpa_cli` (wpasupplicant).

TX Power and Passive
- TX power: `TxPowerSetting` (stealth/low/medium/high/max) sets `iw ... set txpower fixed <mBm>` or falls back to `iwconfig`. Config key `settings.tx_power_level`.
- Passive mode (placeholder UI flow) informs RX-only reconnaissance; actual capture logic lives in `rustyjack-wireless` and is invoked from core operations.

Rustyjack-Wireless Low-Level Notes
- `nl80211.rs` provides `get_mac_address`, `set_mac_address`, channel setting, monitor mode helpers.
- Frame parsing/structs in `frames.rs`; deauth frames crafted in `inject.rs`; PMKID/handshake parsing in `pmkid.rs`, `handshake.rs`.
- Monitor/injection support depends on driver capabilities of the active interface; built-in CYW43436 cannot inject without Nexmon patches.

Installers (install_rustyjack.sh / install_rustyjack_dev.sh)
- Install apt deps: build chain, firmware packs, wireless tools (`iw`, `wireless-tools`), wpa/hostapd/dnsmasq, iproute2, isc-dhcp-client, network-manager, drivers/firmware for USB adapters.
- Enable SPI/I2C and add GPIO pull-ups line to `/boot/firmware/config.txt`, load SPI/I2C modules, ensure swap for builds.
- Build `rustyjack-ui` (release or debug), install to `/usr/local/bin`, enable and start systemd service.
- Create loot and wifi profile directories under project root.

Deep-Dive Architecture and Code Paths
UI Event Loop and Rendering (rustyjack-ui/src/app.rs)
- Core objects on startup: `CoreBridge` (RPC into rustyjack-core), `Display` (ST7735S driver), `ButtonPad` (GPIO input), `GuiConfig` (loaded from gui_conf.json), `MenuTree` (menu definitions), `StatsSampler` (overlay telemetry), and `MenuState` (stack/selection).
- Input mapping (`map_button`) normalizes GPIO events to `ButtonAction`. `choose_from_menu` renders a paged menu with offset handling:
  ```rust
  let displayed_selected = index.saturating_sub(offset);
  self.display.draw_menu(title, &slice, displayed_selected, &overlay)?;
  ```
  Up/Down wrap, Left backs out, Right/Select returns the index, Key1 refreshes the view, Key2 jumps home, Key3 opens reboot confirm.
- Dialogs (`show_message`, `show_progress`) draw overlay (stats) plus content and wait for user dismissal (SELECT/RIGHT or LEFT).

Display Driver Wiring (rustyjack-ui/src/display.rs)
- SPI via `/dev/spidev0.0` with GPIO-controlled lines:
  - DC: `chip.get_line(25)` (BCM25)
  - RST: `chip.get_line(27)` (BCM27)
  - BL: `chip.get_line(24)` (BCM24) for backlight on/off
- Uses `st7735-lcd` + `embedded-graphics` to draw text/rectangles on a 128×128 buffer. Rotation is set by `RUSTYJACK_DISPLAY_ROTATION`.

Buttons and GPIO (rustyjack-ui/src/input.rs)
- `ButtonPad::new(&config.pins)` consumes pin mapping from config defaults (BCM: 6/19/5/26/13/21/20/16). Uses `gpiod` lines with pull-ups (bias) and waits on edge events for low-latency input polling.

Configuration (rustyjack-ui/src/config.rs)
- `GuiConfig` persisted in `gui_conf.json` with sections:
  - `pins` (BCM mappings)
  - `colors` (UI palette)
  - `paths` (image browser start path)
  - `settings`:
    - `active_network_interface`
    - `target_network`, `target_bssid`, `target_channel`
    - `mac_randomization_enabled`, `original_mac`, `current_mac`
    - `tx_power_level`
    - `passive_mode_enabled`
- Defaults reflect Waveshare pinout and Wi-Fi interface `wlan0`.

CoreBridge and Command Dispatch (rustyjack-ui/src/core.rs)
- `CoreBridge::dispatch(cmd: Commands)` is the RPC entry to rustyjack-core. UI methods build `Commands::Wifi(...)` variants with args structs (e.g., `WifiScanArgs`, `WifiDeauthArgs`) and parse JSON `serde_json::Value` responses into typed structs (e.g., `WifiScanResponse`).

Target Usage Across Features
- Deauth: validates `target_bssid` and `target_channel` are set, then dispatches:
  ```rust
  WifiDeauthArgs { bssid, ssid, interface, channel, duration:120, packets:64,
                   client:None, continuous:true, interval:1 }
  ```
- Evil Twin: requires SSID/BSSID/channel/interface, dispatches `WifiEvilTwinArgs { ssid, target_bssid:Some(...), channel, interface, duration:300, open:true }`.
- PMKID Capture: offers targeted or passive:
  ```rust

                  ssid: if use_target { Some(target_network) } else { None },
                  channel: if use_target { target_channel } else { 0 },
                  duration }
  ```
- Probe Sniff: uses active interface, channel hop (`channel: 0`), dispatches `WifiProbeSniffArgs`.
- Handshake cracking and loot reading rely on files in `loot/Wireless`.

MAC Randomization (UI)
- Uses `rustyjack-evasion::MacManager`:
  ```rust
  mac_manager.get_mac(&iface)?;
  mac_manager.set_with_strategy(&iface, MacGenerationStrategy::Vendor(vendor.name))
  // fallback: mac_manager.randomize(&iface)
  ```
- Saves `original_mac` and `current_mac` to config. After change, runs `dhclient -r` then `dhclient`, and signals reconnect via `wpa_cli reconnect` or `nmcli device reconnect`. Restore uses `ip link set <iface> address <original>`.
- Auto-randomize (`auto_randomize_mac_if_enabled`) mirrors the same manager flow.

Tx Power Control (rustyjack-ui/src/app.rs)
- Maps `TxPowerSetting` to dBm and attempts `iw dev <iface> set txpower fixed <mBm>`; if that fails, falls back to `iwconfig <iface> txpower <dBm>`. Messages reflect capability errors.

Passive Mode (UI placeholder)
- Presents “Passive Recon” durations but currently displays a simulated message; real passive capture is handled in `rustyjack-wireless` and would be invoked via core commands when implemented.

rustyjack-core (selected operations)
- `src/evasion.rs`: MAC generation (`generate_random_mac`), rotation (`start_mac_rotation` thread), TTL spoofing, packet fragmentation, OS fingerprint tweaks via sysctl, and MAC set/get helpers (using `ip link`).
- CLI wiring (not detailed in this file) routes `WifiCommand` variants to implementations that wrap `rustyjack-wireless` for scan, deauth, PMKID, profiles, etc., and returns JSON maps consumed by the UI.

rustyjack-evasion (MAC and TX utilities)
- `mac.rs`: `MacManager` (tracks original/current MAC, secure RNG via `getrandom`, enforces locally administered/unicast bits), `MacGenerationStrategy` (Random, Vendor, CustomOui, Specific), interface validation (exists, no '/' or NUL), down/up helpers using `ip link`.
- `vendor.rs`: OUI database and lookups (`from_name`, `from_oui`); used to blend randomized MACs with vendor prefixes (e.g., Apple, Intel).
- `txpower.rs` (not detailed above) manages TX power settings via `iw`/`iwconfig`.

rustyjack-wireless (802.11 low-level)
- `nl80211.rs`: helpers for MAC get/set (`/sys/class/net/<iface>/address`), interface state, channel setting, monitor checks. `set_mac_address` brings interface down, sets MAC with `ip`, brings it up.
- `frames.rs`: defines `MacAddress` and frame parsing/formatting utilities.
- `inject.rs`: crafts deauth frames and uses monitor-capable interfaces to inject.
- `pmkid.rs`, `handshake.rs`: parse PMKID and 4-way handshake messages from captured frames.
- Capability requirement: monitor + injection support from the active interface/driver; built-in CYW43436 cannot meet this without Nexmon patches.

```rust
pub fn random() -> Result<Self> {
    let mut bytes = [0u8; 6];
    getrandom::getrandom(&mut bytes)?;
    bytes[0] = (bytes[0] | 0x02) & 0xFE; // local, unicast
    Ok(Self { bytes })
}
pub fn set_mac(&mut self, interface: &str, mac: &MacAddress) -> Result<MacState> {
    self.interface_down(interface)?;

    result?; up_result?;

}

    let output = Command::new("ip")

}
```

```rust
pub fn get_mac_address(name: &str) -> Result<[u8; 6]> {
    let path = format!("/sys/class/net/{}/address", name);
    let mac_str = fs::read_to_string(&path)?;
    // parse AA:BB:CC:DD:EE:FF
}
pub fn set_mac_address(name: &str, mac: &[u8; 6]) -> Result<()> {
    set_interface_state(name, false)?; // down
    Command::new("ip").args(["link", "set", name, "address", &mac_str]).status()?;
    set_interface_state(name, true)?;  // up
    Ok(())
}
```

Pi Zero 2 W Built-in Wi-Fi Detailed Notes
- CYW43436 (brcmfmac driver): supports managed/AP (2.4 GHz). Does not expose monitor or injection via upstream driver. For deauth/injection/handshake capture, use a USB adapter with monitor+inject (ath9k_htc, rtl8812au, mt76 variants).
- Max PHY: 72.2 Mbps (HT20, 1x1). Channels 1-13 (14 in JP). No 5 GHz or 40 MHz. Channel setting in attacks should remain within 1-13 for the built-in radio; external dual-band adapters extend to 5 GHz.

End-to-End Flow Example (Deauth)
1) User scans networks → selects SSID → sets target → `gui_conf.json` stores SSID/BSSID/channel.
2) User launches Deauth → UI validates target fields and interface.
3) UI displays progress, spawns thread calling `CoreBridge::dispatch(Deauth)`.
4) `rustyjack-core` handles `WifiDeauthArgs` and calls into `rustyjack-wireless` to set monitor mode, set channel, craft deauth frames, and inject.
5) Results (handshake_captured, packets, bursts, log path) returned as JSON → UI shows summary and loot path.

End-to-End Flow Example (MAC Randomize)
1) User taps Randomize MAC → UI reads current MAC with `MacManager::get_mac`.
2) Chooses vendor-aligned MAC if OUI lookup succeeds, else random local-admin MAC.
3) `MacManager` brings iface down, sets MAC via `ip`, brings iface up; records original/current in memory.
4) UI saves `original_mac`/`current_mac` to `gui_conf.json`.
5) UI triggers `dhclient -r && dhclient` and `wpa_cli reconnect` or `nmcli device reconnect` to ensure new MAC is used for layer 2/3.
